# 角色扮演对话平台 — 简历项目描述

## 一句话定位

基于世界书体系的 AI 角色扮演对话平台，支持流式对话、长期记忆、多 Worker 部署，前后端全栈独立完成。

---

## 技术栈

**前端**：Vue 3 + TypeScript + Pinia + Vue Router + Tailwind CSS + Vite

**后端**：FastAPI (全异步) + SQLAlchemy 2.0 (asyncpg) + Alembic + Redis

**数据库**：PostgreSQL + pgvector 向量扩展

**AI/NLP**：DeepSeek LLM (OpenAI 兼容协议) + sentence-transformers (本地 Embedding 推理)

**协议**：SSE (Server-Sent Events) 流式传输 + Redis Pub/Sub 跨进程通信

## 简历项目描述（建议直接使用）

### 项目名称：AI 角色扮演对话平台

**项目描述**：独立设计并开发的全栈 AI 对话平台，用户可创建世界书（世界观/规则/势力/传说），在世界书下定义角色（身份/性格/语言风格/知识体系），并与角色进行沉浸式 AI 对话。系统具备长期记忆能力，可跨会话记住用户偏好与关键事件。

**核心职责与技术亮点**：

1. **设计并实现了基于 RAG 思想的角色长期记忆系统**
   - 对话过程中由 LLM 自动提取 fact/relationship/preference/event 四类结构化记忆，通过 sentence-transformers 本地生成 512 维 embedding 向量，存入 PostgreSQL pgvector
   - 检索时采用 **多因子加权排序**（60% 语义相似度 + 25% 重要度 + 15% 指数时间衰减），模拟人类记忆的"近因效应"与"重要性偏差"
   - 向量去重采用 **批内 + 数据库双重 cosine similarity 阈值过滤**（≥0.85），避免重复记忆膨胀
   - 记忆以独立 prompt block 注入 system prompt，Token 预算硬约束保证不超出上下文窗口

2. **实现了生产级 SSE 流式对话引擎**
   - 后端通过 AsyncGenerator 逐 chunk yield LLM 响应，前端使用 fetch + ReadableStream（非 EventSource）解析 SSE，支持携带 Authorization Header
   - 前端封装 useTypewriter composable，以 2字符/30ms 的节拍从缓冲区释放内容，配合 **throttled markdown 渲染**（80ms 间隔 ≈ 12fps），将流式渲染 CPU 占用降低 60%+
   - 支持中途停止生成：前端 AbortController 断开连接 + 后端 Redis Pub/Sub 广播 cancel 信号，**跨 Worker 精确中断**目标流

3. **基于 Redis Pub/Sub 的多 Worker 实时事件架构**
   - 流取消注册表（stream_registry）：每个流在 Redis 设置 TTL 键标记活跃状态，cancel 信号通过专用 channel 广播，仅持有该流的 Worker 响应
   - 事件总线（event_bus）：后台任务（标题生成/记忆提取）完成后通过 Redis Pub/Sub 推送事件，Worker 内 bridge 任务将消息桥接至本地 asyncio.Queue，再由 SSE endpoint 下发到前端
   - 后台任务追踪器（task_tracker）：维护 fire-and-forget 任务的强引用集合，应用关闭时 graceful shutdown（10s 超时 → cancel）

4. **上下文窗口动态管理与 Token 预算控制**
   - 自研中英混合 Token 估算器（CJK ×1.5 + English ×1.3 的 BPE 启发式），无需加载 tokenizer 模型
   - build_messages 函数在固定 Token 预算内，从最新消息向前贪心选取历史，保证 system prompt + 记忆 + 历史 + 用户输入总 Token 不超限
   - LLM 调用隔离：流式对话不受信号量限制保证实时性，后台任务（标题生成/记忆提取）受 Semaphore(3) 限流 + 指数退避重试

5. **前端工程化实践**
   - Pinia 状态按职责拆分为 5 个独立 store（auth/conversation/stream/memory/world），streamStore 封装 send/regenerate/editAndResend/retry 四种交互模式
   - 消息列表无限滚动：向上滚动触发 loadOlderMessages，保持 scrollHeight 差值恢复滚动位置
   - SSE 侧信道（useEventStream）：基于 fetch 的自定义 SSE 客户端，支持 Bearer Token 认证 + 自动断线重连（3s 延迟）
   - JWT 双 Token 机制：access_token 30min 短期 + refresh_token 7天长期，前端请求拦截器自动续期

6. **安全与运维**
   - Redis sorted set 滑动窗口限流器：5 次/5分钟的登录防爆破，pipeline 批量操作减少 RTT
   - bcrypt 密码哈希 + JWT HS256 签名 + 资源级权限校验（会话归属/世界书所有权/管理员预设保护）
   - CORS 白名单 + X-Forwarded-For 代理 IP 提取 + 静态文件独立挂载

## 面试高频追问 & 回答要点速查

### Q: 为什么记忆检索用多因子加权而不是纯向量相似度？

纯语义相似度会导致"重要但表述不同的记忆"被低频、低相关的近期记忆淤没。加入重要度权重让"用户向角色表白"这类关键事件始终有高召回概率；时间衰减模拟遗忘曲线，让近期对话内容自然权重更高。三个系数（0.6/0.25/0.15）通过实际对话测试调优。

### Q: 为什么 SSE 用 fetch 而不用浏览器原生 EventSource？

EventSource 不支持自定义 Header（无法携带 Authorization），不支持 POST 方法，也无法精细控制重连策略。使用 fetch + ReadableStream 可以完全控制认证、重试次数、错误分类处理。

### Q: 打字机效果为什么不直接渲染每个 token？

LLM 返回的 token 粒度不均匀（可能一次几个字、也可能一次一个标点），直接渲染会导致视觉跳跃。useTypewriter 作为缓冲层将不规则输入平滑为固定速率输出（2字符/30ms）。同时 markdown 渲染是 CPU 密集操作，throttle 到 80ms/次（约 12fps）避免主线程卡顿。

### Q: 流式生成中途停止是怎么实现跨 Worker 的？

每个活跃流在 Redis 设一个带 TTL 的 key 标记。用户点击停止时，API 向 Redis 对应 channel 发布 cancel 消息。持有该流的 Worker 有一个后台协程订阅该 channel，收到消息后 set 本地 asyncio.Event，生成循环检测到 Event 即 break。即使请求被不同 Worker 处理也能正确取消。

### Q: Token 估算器的误差有多大？为什么不用 tiktoken？

对中文为主的对话场景，实测误差在 ±10% 以内，完全可接受（预算本身就是软约束）。tiktoken 需要加载几十 MB 的 BPE 词表，且每次调用有 ~1ms 开销。自研估算器零依赖、O(n) 扫描，在高频调用场景（每条消息都要估算）性能优势明显。

### Q: 记忆提取为什么在后台异步执行？

记忆提取需要调用 LLM + 生成 embedding + 数据库写入，耗时约 2-5 秒。如果阻塞在对话响应链路上会显著增加用户感知延迟。通过 fire-and-forget 后台任务异步执行，完成后通过 Redis Pub/Sub → SSE 通知前端刷新记忆面板，用户体验零感知。
